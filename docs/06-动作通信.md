# 动作通信（Action）

## 为什么需要动作通信

学完了话题（广播）和服务（问答），来看一个它们都搞不定的场景：

**让机器人从 A 点开到 B 点，导航过程中每秒报告一次「还剩多少米」。**

| 方案 | 为什么不行 |
|------|-----------|
| Topic | 单向广播，没法一对一返回结果 |
| Service | 可以一对一，但导航要跑 30 秒——客户端干等 30 秒啥也干不了 |

你需要的是 **Action**：

```
🤖 机器人                                        💻 用户
┌──────────┐         🎯 目标：从 A 开到 B         ┌──────────┐
│          │  ←──────────────────────────────    │          │
│          │         反馈：还剩 20m               │          │
│  Action  │  ──────────────────────────────→    │  Action  │
│  Server  │         反馈：还剩 10m               │  Client  │
│          │  ──────────────────────────────→    │          │
│          │         反馈：还剩 5m                │          │
│          │  ──────────────────────────────→    │          │
│          │         ✅ 结果：到达！              │          │
│          │  ──────────────────────────────→    │          │
└──────────┘                                     └──────────┘
```

动作通信 = **目标 + 反馈 + 结果 + 可中途取消**。

---

## 三大通信模型对比

```
话题 Topic         服务 Service          动作 Action
─────               ─────                 ─────
单向广播           一问一答               目标→反馈→结果
持续发送           一次调用即结束          可能很久才结束
不需要回复         必须等回复              收到结果前可以收多次反馈
不能取消           不能取消               ✅ 可以中途取消
收音机 📻          打电话 📞              外卖下单 🛵
```

| | Topic | Service | Action |
|------|-------|---------|--------|
| 通信方向 | 单向（pub→sub） | 双向（req⇄res） | 双向（goal→feedback→result） |
| 调用时长 | 持续 | 瞬时 | 可能很久 |
| 中间反馈 | 无 | 无 | **有（反馈）** |
| 可取消 | 否 | 否 | **是** |
| 适用场景 | 传感器数据流 | 查询、配置、指令 | 导航、抓取、长时间计算 |
| 客户端阻塞 | 不阻塞 | 阻塞等待响应 | 异步，收反馈不阻塞 |

## 动作的定义：.action 文件

```
# goal（目标，客户端发）
int32 order      # 要算多少位斐波那契数列
---
# result（结果，服务端最后返回）
int32[] sequence  # 完整的斐波那契数列
---
# feedback（反馈，服务端中间进度）
int32[] partial_sequence  # 到目前为止算出的部分结果
```

三部分，两个 `---` 隔开：**目标 → 结果 → 反馈**。

和 `.srv` 不同，Action 的请求不叫 request，叫 **goal**（目标）——「我要达到什么」。服务端一边执行一边发 **feedback**（进度），最终给 **result**（结果）。

> ROS2 内置的 `example_interfaces` 包里有 `Fibonacci`，正好拿来做第一个动作通信例子，不用自己写 `.action` 文件。

## 动作服务端（Action Server）

服务端的工作是：接收目标 → 干活 → 发反馈 → 返回结果。`learning_action/fibonacci_action_server.py`：

```python
"""斐波那契动作服务端

提供 fibonacci 动作：接受 order=N，计算 N 位斐波那契数列。
每算 1 位就发送一次反馈（部分序列），支持中途取消。

运行方式：
  ros2 run learning_action fibonacci_action_server
"""

import time
from example_interfaces.action import Fibonacci
import rclpy
from rclpy.node import Node
from rclpy.action import ActionServer, CancelResponse, GoalResponse
from rclpy.callback_groups import ReentrantCallbackGroup


class FibonacciActionServer(Node):
    """斐波那契动作服务端"""

    def __init__(self):
        super().__init__('fibonacci_action_server')

        # ① 创建 ActionServer
        #    参数：node、动作类型、动作名、四个回调
        self._action_server = ActionServer(
            self,
            Fibonacci,
            'fibonacci',
            goal_callback=self.goal_callback,
            cancel_callback=self.cancel_callback,
            execute_callback=self.execute_callback,
            callback_group=ReentrantCallbackGroup(),  # 允许取消回调打断执行回调
        )
        self.get_logger().info('🤖 动作服务端已启动，等待目标…')

    # ━━━ 三个回调 ━━━

    def goal_callback(self, goal_request):
        """② 收到新目标时立即调用——接不接这个活？

        goal_request.order = 客户端要算多少位

        返回：
          GoalResponse.ACCEPT  —— 接了
          GoalResponse.REJECT  —— 不接（比如参数不合法）
        """
        order = goal_request.order
        self.get_logger().info(f'📥 收到目标: order={order}')

        if order <= 0:
            self.get_logger().warn(f'❌ order 必须 > 0，拒绝')
            return GoalResponse.REJECT

        return GoalResponse.ACCEPT

    def cancel_callback(self, goal_handle):
        """③ 客户端请求取消时调用——让不让取消？

        几乎总是回答 ACCEPT。你可以在这里做清理工作。
        """
        self.get_logger().info(f'🛑 收到取消请求')
        return CancelResponse.ACCEPT

    def execute_callback(self, goal_handle):
        """④ 干活的地方：算斐波那契数列。

        这是 ActionServer 的核心回調。它在一个独立线程里跑，
        所以可以用 time.sleep() 模拟耗时操作。

        goal_handle 提供了三个关键方法：
          - publish_feedback(feedback)    发中间进度
          - succeed()                     宣告成功
          - abort()                       宣告失败
          - canceled()                    宣告被取消
          - is_cancel_requested           是否有人要求取消？
        """
        order = goal_handle.request.order
        self.get_logger().info(f'⚙️  开始计算: order={order}')

        # 初始化反馈对象
        feedback_msg = Fibonacci.Feedback()
        sequence = [0, 1]

        for i in range(1, order):
            # ── 计算下一项 ──
            sequence.append(sequence[i] + sequence[i - 1])

            # ── 发反馈 ──
            feedback_msg.partial_sequence = sequence
            goal_handle.publish_feedback(feedback_msg)
            self.get_logger().info(f'📊 反馈: {sequence}')

            # ── 模拟耗时 ──
            time.sleep(1.0)

            # ── 检查是否被取消 ──
            if goal_handle.is_cancel_requested:
                self.get_logger().info('🛑 任务已取消')
                goal_handle.canceled()                  # 宣告取消
                return Fibonacci.Result()                # 返回空结果

        # ── 计算完成，返回最终结果 ──
        goal_handle.succeed()                            # 宣告成功

        result = Fibonacci.Result()
        result.sequence = sequence
        self.get_logger().info(f'✅ 计算完成: {sequence}')
        return result


def main():
    rclpy.init()
    node = FibonacciActionServer()
    rclpy.spin(node)
    node.destroy_node()
    rclpy.shutdown()


if __name__ == '__main__':
    main()
```

### 三个回调，三种角色

| 回调 | 调用时机 | 作用 | 返回值 |
|------|---------|------|--------|
| `goal_callback` | 收到新目标时 | 接不接这个活？ | `ACCEPT` / `REJECT` |
| `cancel_callback` | 客户端请求取消时 | 让不让取消？ | `ACCEPT` |
| `execute_callback` | goal 被接受后 | **真正干活** | `Result` 对象 |

### execute_callback 的五个关键操作

```
goal_handle.publish_feedback(feedback)   发进度
goal_handle.is_cancel_requested          检查取消请求
goal_handle.succeed()                   宣告成功
goal_handle.canceled()                  宣告已取消
goal_handle.abort()                     宣告异常终止（出错了）
```

> `succeed()` / `canceled()` / `abort()` 三选一，不调的话客户端永远收不到结果。

### ReentrantCallbackGroup 是什么

```python
from rclpy.callback_groups import ReentrantCallbackGroup

self._action_server = ActionServer(
    ...,
    callback_group=ReentrantCallbackGroup(),
)
```

默认情况下 ROS2 的回调是**互斥**的——同一时间只能跑一个回调。但 Action 有个特殊需求：`execute_callback` 正在跑的时候，客户端可能会发取消请求，这时 `cancel_callback` 需要**中断**当前执行。

`ReentrantCallbackGroup` 允许多个回调同时跑，这样取消请求才能及时处理。

## 动作客户端（Action Client）

客户端的工作是：发目标 → 收反馈 → 收结果。`learning_action/fibonacci_action_client.py`：

```python
"""斐波那契动作客户端

向 fibonacci 动作服务端发送目标，接收反馈并打印最终结果。

运行方式：
  ros2 run learning_action fibonacci_action_client         # 默认 order=10
  ros2 run learning_action fibonacci_action_client 5       # order=5
"""

import sys
from example_interfaces.action import Fibonacci
import rclpy
from rclpy.node import Node
from rclpy.action import ActionClient


class FibonacciActionClient(Node):
    """斐波那契动作客户端"""

    def __init__(self):
        super().__init__('fibonacci_action_client')

        # ① 创建 ActionClient：动作类型 + 动作名
        self._action_client = ActionClient(self, Fibonacci, 'fibonacci')

        self.get_logger().info('⏳ 等待动作服务端…')
        self._action_client.wait_for_server()
        self.get_logger().info('✅ 已连接动作服务端')

    def send_goal(self, order):
        """② 发送目标，注册反馈回调"""
        goal_msg = Fibonacci.Goal()
        goal_msg.order = order

        # send_goal_async 返回一个 Future
        self._send_goal_future = self._action_client.send_goal_async(
            goal_msg,
            feedback_callback=self.feedback_callback  # 反馈来了就调这个
        )

        # future 完成时 → goal_response_callback 被调用
        self._send_goal_future.add_done_callback(self.goal_response_callback)

    # ━━━ 两个回调 ━━━

    def goal_response_callback(self, future):
        """③ 服务端告诉我们目标被接受还是拒绝"""
        goal_handle = future.result()

        if not goal_handle.accepted:
            self.get_logger().info('❌ 目标被拒绝')
            return

        self.get_logger().info('✅ 目标已接受，正在执行…')

        # 等待最终结果
        self._get_result_future = goal_handle.get_result_async()
        self._get_result_future.add_done_callback(self.get_result_callback)

    def feedback_callback(self, feedback_msg):
        """④ 每次收到反馈时调用（中间进度）"""
        seq = feedback_msg.feedback.partial_sequence
        self.get_logger().info(f'📊 反馈: {seq}')

    def get_result_callback(self, future):
        """⑤ 收到最终结果时调用"""
        result = future.result().result
        self.get_logger().info(f'🎉 最终结果: {result.sequence}')
        rclpy.shutdown()


def main():
    rclpy.init()
    node = FibonacciActionClient()

    order = int(sys.argv[1]) if len(sys.argv) >= 2 else 10
    node.send_goal(order)

    rclpy.spin(node)


if __name__ == '__main__':
    main()
```

### 客户端调用链：一看就懂的异步链条

```
send_goal(order)
    │
    ├─→ 服务端收到 goal，调 goal_callback(ACCEPT/REJECT)
    │
    ├─→ goal_response_callback(future)
    │       │
    │       ├─ 被拒绝 → 打印 + 结束
    │       │
    │       └─ 被接受 → goal_handle.get_result_async()
    │                       │
    │                       └─→ get_result_callback(future)  拿到最终结果
    │
    └─→ feedback_callback(msg) ← 每收到一次反馈就调一次
```

和服务客户端不同，Action 客户端**不阻塞**——反馈随时到随时处理，最终结果到了再用回调收。

## 为什么 ActionServer 不像 Service 那样阻塞

| | Service | Action |
|------|---------|--------|
| 回调执行位置 | spin 线程 | **独立线程** |
| 能否 `time.sleep()` | ❌ 会卡死 spin | ✅ 只影响自己 |
| 需要 ReentrantCallbackGroup？ | 不需要 | **需要** |
| 取消机制 | 无 | `is_cancel_requested` |

ActionServer 的 `execute_callback` 在独立线程里跑，所以你可以放心 `time.sleep(1.0)`。但正因为它独立跑，取消回调才能同时执行——`ReentrantCallbackGroup` 就是为这个设计的。

## 什么时候用 Action

```
✅ 适合用 Action：
  - 导航（「开到 B 点，每秒报告位置」）
  - 长时间计算（「算 100 位圆周率」）
  - 机器人抓取（「移动机械臂→抓→检测→回正」）
  - 需要反馈的任务（「还剩 30%」）
  - 需要中途取消的任务（「停下！别走了！」）

❌ 不需要 Action：
  - 瞬时操作 → 用 Service
  - 持续数据流 → 用 Topic
```

### 决策树

```
         需要回复吗？
        /          \
      否            是
      │             │
      │         耗时久 / 需要反馈 / 可取消？
      │         /          \
    Topic      否           是
               │            │
            Service       Action
```

## 总结

```
动作通信 = 目标(Goal) + 反馈(Feedback) + 结果(Result)

ActionServer 三个回调：
  ① goal_callback      → ACCEPT / REJECT
  ② cancel_callback    → ACCEPT
  ③ execute_callback   → 干活 + 发反馈 → succeed/canceled/abort

ActionClient 四个步骤：
  ① send_goal_async(goal, feedback_callback)   发目标
  ② goal_response_callback                     目标被接受/拒绝
  ③ feedback_callback                         收中间进度（可能多次）
  ④ get_result_callback                       收最终结果

关键差异：
  - Action 是异步的 —— 反馈随时到，不阻塞等结果
  - execute_callback 跑在独立线程 —— 可以 time.sleep
  - 需要 ReentrantCallbackGroup —— 取消请求才能打断执行
  - ros2 action send_goal 是测试利器
```

| 对比 | 话题 Topic | 服务 Service | 动作 Action |
|------|-----------|-------------|-------------|
| 角色 | 发布者/订阅者 | 服务端/客户端 | 服务端/客户端 |
| 创建方式 | `create_publisher` / `create_subscription` | `create_service` / `create_client` | `ActionServer` / `ActionClient` |
| 通信方向 | 单向 | 请求→响应 | 目标→反馈→结果 |
| 可否取消 | 否 | 否 | **是** |
| CLI 工具 | `ros2 topic` | `ros2 service` | `ros2 action` |

话题是「收音机」，服务是「打电话」，动作是「外卖下单」。ROS2 三大通信模型，齐了。
