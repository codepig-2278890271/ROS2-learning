# 工作空间与功能包

## 一、怎么创建功能包

创建功能包用 `ros2 pkg create` 命令，Python 和 C++ 的区别只在于 `--build-type`。

### Python 包

**必须在工作空间的 `src/` 目录下执行：**

```bash
cd ~/ros2_ws/src     # 先进入 src/，必须在这一步

ros2 pkg create --build-type ament_python <包名> \
  --dependencies <依赖1> <依赖2> ... \
  --description "包的描述" \
  --license Apache-2.0
```

> **为什么必须在 `src/` 下？** `colcon build` 只扫描工作空间根目录下的 `src/` 文件夹来寻找功能包。功能包放在别处，`colcon build` 看不到，也就不会编译。

| 参数 | 必选？ | 说明 |
|------|--------|------|
| `--build-type ament_python` | ✅ 必选 | 告诉 ROS2 这是 Python 包，编译时会调用 `setup.py` |
| `<包名>` | ✅ 必选 | 小写字母 + 下划线，如 `learning_node`、`lidar_driver` |
| `--dependencies` | 可选 | 依赖的其他包名，逗号分隔。写了会自动填到 `package.xml`，**忘了后面也可以手动补** |
| `--description` | 可选 | 一句话描述，写进 `package.xml` |
| `--license` | 可选 | 许可证，社区默认 `Apache-2.0` |

### C++ 包

```bash
ros2 pkg create --build-type ament_cmake <包名> \
  --dependencies rclcpp std_msgs
```

| 参数 | 说明 |
|------|------|
| `--build-type ament_cmake` | 告诉 ROS2 这是 C++ 包，编译时会调用 `CMakeLists.txt` |
| `--dependencies rclcpp` | 对应 C++ 客户端库 `rclcpp`（Python 用 `rclpy`） |

> 其他参数和 Python 包一样。

### 最小化创建

最简单的写法，什么都不加：

```bash
ros2 pkg create --build-type ament_python learning_node
```

依赖、描述、许可证全部空着，后面手动改 `package.xml` 补上就行。

## 二、工作空间和功能包是什么关系

### 工作空间

工作空间就是你的**项目根目录**，把散落的包收纳在一起，方便统一编译和管理。

```
ros2_ws/                  # 工作空间（文件夹，名字随意）
├── src/                  # 源码：所有功能包放这里（唯一要 git 管理的）
├── build/                # 编译中间文件（自动生成，可以删）
├── install/              # 编译产物：实际运行时用的文件在这里（自动生成，可以删）
└── log/                  # 编译日志（自动生成，可以删）
```

- `src/` 你是**手写的**
- `build/` `install/` `log/` 都是 `colcon build` **自动生成的**
- 三个自动目录删了无所谓，再跑一次 `colcon build` 就回来了
- 一个工作空间可以放**多个功能包**

### 功能包

功能包是 ROS2 代码的**最小组织单位**。一个包 = 一个独立功能模块。

举个例子，一个机器人项目可能包含这些包：

```
ros2_ws/src/
├── lidar_driver/         # 激光雷达驱动
├── camera_driver/        # 摄像头驱动
├── slam_node/            # 建图算法
├── path_planner/         # 路径规划
└── motor_controller/     # 电机控制
```

### 关系一句话总结

> **工作空间是容器，功能包是内容。** 打包放进容器里，`colcon build` 一次性全部编译。

## 三、创建和编译后，各文件干什么用

以 Python 包为例，`ros2 pkg create --build-type ament_python learning_node` 生成的目录结构：

```
learning_node/              ← ① ROS2 功能包目录（外层）
├── package.xml              # 身份证
├── setup.py                 # 安装脚本
├── setup.cfg                # 安装路径
├── resource/
│   └── learning_node        # 空标记文件
├── test/
│   ├── test_copyright.py    # 样板测试（可以删）
│   ├── test_flake8.py       # 样板测试（可以删）
│   └── test_pep257.py       # 样板测试（可以删）
└── learning_node/           ← ② Python 包目录（内层，和包同名）
    └── __init__.py          # 标识这是一个 Python 包
```

### 为什么有两层同名目录？

这是新手最容易困惑的地方，但理解了就很简单——两层目录职责不同：

```
外层 learning_node/          ROS2 功能包：放元信息文件（package.xml、setup.py…）
  ├── package.xml            ← ROS2 才关心的东西
  ├── setup.py               ← ROS2 才关心的东西
  │
  └── 内层 learning_node/    Python 包：放实际代码（节点脚本、算法…）
       ├── __init__.py       ← Python 才关心的东西
       ├── publisher.py      ← 你写的代码
       └── subscriber.py     ← 你写的代码
```

| | 外层目录 | 内层同名目录 |
|------|---------|------------|
| 是什么 | ROS2 功能包 | Python 包 |
| 谁看 | `colcon build`、`ros2 pkg` | `import` 语句、Python 解释器 |
| 放什么 | `package.xml`、`setup.py`、`setup.cfg`、`resource/` | `__init__.py`、节点脚本、自定义模块 |

C++ 包就没有这个内层同名目录，因为 C++ 不需要 Python 的 `import` 机制。这是 Python 包特有的结构。

> **为什么两层必须同名？** `setup.py` 里通过这个名字关联二者——`ros2 run` 根据命令名找到外层包，再根据 `entry_points` 定位到内层 Python 模块的某个函数。

### package.xml — 身份证

告诉 ROS2：你是谁、依赖谁、谁维护。

```xml
<package format="3">
  <name>learning_node</name>        <!-- 包名，必须和目录名一致 -->
  <version>0.0.0</version>          <!-- 版本号 -->
  <description>xxx</description>         <!-- 一句话描述 -->
  <maintainer email="xxx">xxx</maintainer> <!-- 维护者 -->
  <license>Apache-2.0</license>          <!-- 许可证 -->

  <depend>rclpy</depend>            <!-- 依赖的包，可以写多个 -->
  <depend>std_msgs</depend>

  <test_depend>ament_copyright</test_depend>  <!-- 仅测试用 -->
  <test_depend>ament_flake8</test_depend>
  <test_depend>ament_pep257</test_depend>
  <test_depend>python3-pytest</test_depend>

  <export>
    <build_type>ament_python</build_type>    <!-- 声明：这是 Python 包 -->
  </export>
</package>
```

**依赖标签一览：**

| 标签 | 什么时候用 |
|------|-----------|
| `<depend>` | **最常用**，编译和运行时都需要 |
| `<build_depend>` | 仅编译时需要 |
| `<exec_depend>` | 仅运行时需要 |
| `<test_depend>` | 仅测试时需要 |

> 初学记住 `<depend>` 就够了，忘了加依赖就手动补一行，改完重新 `colcon build` 即可。

### setup.py — 安装脚本

`colcon build` 本质上就是执行 `python setup.py install`。它决定了：

1. 哪些 Python 文件要安装
2. 哪些非 Python 文件（package.xml、resource 标记）要复制到 `install/`
3. 你的节点叫什么名字、对应哪个 Python 函数（`entry_points`）

```python
from setuptools import find_packages, setup

package_name = 'learning_node'

setup(
    name=package_name,
    version='0.0.0',
    packages=find_packages(exclude=['test']),
    data_files=[
        ('share/ament_index/resource_index/packages',
            ['resource/' + package_name]),
        ('share/' + package_name, ['package.xml']),
    ],
    install_requires=['setuptools'],
    zip_safe=True,
    maintainer='xxx',
    maintainer_email='xxx@xxx.com',
    description='xxx',
    license='Apache-2.0',
    tests_require=['pytest'],
    entry_points={
        'console_scripts': [
            # 重点在这里：注册可执行命令
            # '命令名 = learning_node.文件名:main函数名',
        ],
    },
)
```

**最核心的两个部分：**

**`data_files`** — 告诉 colcon 把非 Python 文件复制到 `install/` 的哪个位置：
- `resource/learning_node` → 标记文件，让 `ros2 pkg list` 能找到你的包
- `package.xml` → 也要装过去，方便工具链读取元信息

**`entry_points`** — 注册你的可执行命令：

```python
entry_points={
    'console_scripts': [
        'talker = learning_node.publisher:main',
    ],
},
```

这行表示：当用户执行 `ros2 run learning_node talker` 时，去 `learning_node/publisher.py` 里找 `main()` 函数来运行。

> 格式：`'命令名 = 包名.文件名:函数名'`

### setup.cfg — 安装路径

告诉 setuptools 把可执行文件装到哪：

```ini
[develop]
script_dir=$base/lib/learning_node
[install]
install_scripts=$base/lib/learning_node
```

初学不用改，知道有它就行。

### resource/learning_node — 空标记文件

内容是空的，文件名必须和包名一致。作用：让 ROS2 的索引系统（`ament_index`）能找到这个包。没有它，`ros2 pkg list` 就看不到你的包。

> 类比：图书馆索书号。书本身不需要它，但图书馆靠它找到书。

### learning_node/__init__.py — Python 包标识

内容为空。告诉 Python：「这个目录是一个 Python 包，可以 `import learning_node`」。

你的节点代码就写在这个目录下，比如 `learning_node/publisher.py`、`learning_node/subscriber.py`。

### test/ — 自动生成的测试

三个样板测试文件，分别检查版权声明、代码风格、文档规范。**初学阶段可以暂时不管，甚至删掉**，后面学会了再写正经测试。

### colcon build 后多了什么

```bash
cd ~/ros2_ws
colcon build --symlink-install
```

编译完后，`ros2_ws/` 下会多出三个目录：

| 目录 | 内容 |
|------|------|
| `build/` | 编译中间文件。比如把你的 Python 文件、package.xml 收集到一起，准备安装 |
| `install/` | 编译最终产物。`ros2 run` 实际执行的就是这里的文件 |
| `log/` | 编译过程的日志。编译出错了来这里翻原因 |

**每次编译后要 source：**

```bash
source ~/ros2_ws/install/setup.bash
```

这会让当前终端「认识」你刚编译好的包，`ros2 run learning_node xxx` 才能找到。

**--symlink-install 是什么？**

```
不加：把文件复制到 install/  →  改了源码要重新 colcon build
加了：创建符号链接指向 src/  →  改 Python 代码不用重新编译，直接 ros2 run 生效
```

> 初学必用 `--symlink-install`。只要不改 `package.xml` 和 `setup.py`，改 Python 源码立即生效，省掉无数次重新编译。

**选择性编译：--packages-select vs --packages-up-to**

当你 `src/` 下有多个包，只想编译其中一个时：

```bash
colcon build --packages-select <包名>   # 只编译这一个包
colcon build --packages-up-to <包名>    # 编译这个包 + 它依赖的所有包
```

举个例子，假设三个包有依赖关系：

```
motor_controller  ←── 依赖 —— path_planner  ←── 依赖 —— lidar_driver
（控制电机）              （路径规划）              （雷达驱动）
```

| 命令 | 效果 | 适用场景 |
|------|------|---------|
| `--packages-select path_planner` | 只编译 path_planner，其他跳过 | 快速验证，确定依赖没变 |
| `--packages-up-to path_planner` | 从 lidar_driver 编译到 path_planner，共两个 | 比全量快，比 select 稳 |
| `colcon build`（不加参数） | 全部编译 | 不知道改了啥，全量最省心 |

其他常用参数：

| 命令 | 效果 |
|------|------|
| `--packages-skip <包名>` | 编译所有，**跳过**指定包 |
| `--packages-start <包名>` | 从指定包开始，**往后**编译（不编它的依赖） |

> 新手建议：包不多的时候直接 `colcon build` 全量编译，别纠结。包多了、编译慢了，用 `--packages-up-to` 是最稳妥的选择。

## 总结

```
创建包：ros2 pkg create --build-type ament_python <包名> --dependencies rclpy

工作空间             功能包
┌────────┐         ┌──────────────┐
│ ros2_ws│  ⸧  包含  │ learning_node│  包含  ┌ package.xml  ← 身份证（名称、依赖）
│  src/  │────────→│    ...       │──────→│ setup.py     ← 安装入口 + 注册命令
└────────┘         └──────────────┘        │ setup.cfg    ← 路径配置
                                           │ resource/    ← 标记文件
                                           │ learning_node│ ← 代码写这里
                                           └ test/        ← 测试（可删）
```
