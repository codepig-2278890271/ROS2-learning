# 服务通信（Service）

## 什么是服务通信

话题通信是**单向广播**——发布者只管发，不知道谁在收、有没有收到。但很多时候你需要**一问一答**：

```
客户端（Client）                    服务端（Server）
┌─────────────┐                    ┌─────────────┐
│  图像处理    │  ── 请求 ──→      │  相机驱动    │
│  Client     │  ←── 响应 ──      │  Server     │
└─────────────┘                    └─────────────┘

请求：「分辨率是多少？」
响应：「1920×1080」
```

- 客户端**发送请求（request）**，服务端处理后**返回响应（response）**
- 请求和响应的数据结构由 `.srv` 文件定义
- 一次调用 = 一次往返，调用期间客户端**阻塞等待**（也可以异步非阻塞）

对比话题通信：

| | 话题 Topic | 服务 Service |
|------|-----------|-------------|
| 方向 | 单向（发布 → 订阅） | 双向（请求 ⇄ 响应） |
| 发起方 | 发布者主动发 | 客户端主动请求 |
| 响应 | 不需要等待回复 | **必须等待回复** |
| 适用场景 | 持续的数据流（传感器、状态） | 离散的指令（查询、配置、计算） |
| 类比 | 收音机广播 | 打电话 |

---

## 服务定义：.srv 文件

服务接口用 `.srv` 文件定义，分上下两半，用 `---` 隔开：

```
# 上半：请求（客户端发过来的）
int64 a
int64 b
---
# 下半：响应（服务端返回去的）
int64 sum
```

这条定义的意思是：客户发两个 `int64` 整数，服务回一个 `int64` 整数（两数之和）。

ROS2 内置了一些常用服务类型，放在 `example_interfaces` 包。其中 `AddTwoInts` 就是上面这个定义——正好拿来学服务通信，不用自己写 `.srv` 也不用折腾编译配置。

---

## 服务端：最小示例

服务端的工作是「等请求 → 处理 → 返回结果」。`learning_service/learning_service/add_two_ints_server.py`：

```python
import rclpy
from rclpy.node import Node
from example_interfaces.srv import AddTwoInts          # ① 导入服务类型


class AddTwoIntsServer(Node):
    def __init__(self):
        super().__init__('add_two_ints_server')

        # ② 创建服务端：服务类型 + 服务名 + 回调函数
        self.srv = self.create_service(
            AddTwoInts, 'add_two_ints', self.add_two_ints_callback
        )
        self.get_logger().info('🔧 服务端已启动，等待请求…')

    def add_two_ints_callback(self, request, response):
        """③ 收到请求时自动调用，request 是客户端发来的，response 是要返回的"""
        response.sum = request.a + request.b
        self.get_logger().info(f'收到: {request.a} + {request.b} = {response.sum}')
        return response                                 # ④ 返回 response 给客户端


def main():
    rclpy.init()
    node = AddTwoIntsServer()
    rclpy.spin(node)                                    # ⑤ spin 等待请求
    node.destroy_node()
    rclpy.shutdown()
```

### 逐行解释

```python
from example_interfaces.srv import AddTwoInts
```

`AddTwoInts` 是 ROS2 内置的服务类型，定义就是上面那个 `.srv`：请求里 `a` 和 `b` 两个 `int64`，响应里 `sum` 一个 `int64`。

```python
self.srv = self.create_service(AddTwoInts, 'add_two_ints', self.add_two_ints_callback)
```

三个参数：

| 参数 | 值 | 含义 |
|------|-----|------|
| `AddTwoInts` | 服务类型 | 规定了请求和响应的数据结构 |
| `'add_two_ints'` | 服务名 | 客户端通过这个名字找到服务端 |
| `callback` | 回调函数 | 收到请求时自动调用 |

```python
def add_two_ints_callback(self, request, response):
    response.sum = request.a + request.b
    return response
```

和话题回调不同，服务回调**有入参也有返回值**：

- `request` —— 客户端发来的数据，`request.a`、`request.b` 就是客户端传来的两个数
- `response` —— 要返回给客户端的数据，往 `response.sum` 里填结果
- `return response` —— 把结果送回客户端

## 客户端：最小示例

`learning_service/learning_service/add_two_ints_client.py`：

```python
import rclpy
from rclpy.node import Node
from example_interfaces.srv import AddTwoInts


class AddTwoIntsClient(Node):
    def __init__(self):
        super().__init__('add_two_ints_client')

        # ① 创建客户端：服务类型 + 服务名
        self.cli = self.create_client(AddTwoInts, 'add_two_ints')

        # ② 等待服务端上线（服务端没启动的话请求会失败）
        while not self.cli.wait_for_service(timeout_sec=1.0):
            self.get_logger().info('⏳ 等待服务端上线…')
        self.get_logger().info('✅ 服务端已连接')

    def send_request(self, a, b):
        """③ 发送请求，返回 future 对象"""
        request = AddTwoInts.Request()                  # 构造请求
        request.a = a
        request.b = b
        self.future = self.cli.call_async(request)      # ④ 异步调用
        return self.future


def main():
    rclpy.init()
    node = AddTwoIntsClient()

    # ⑤ 发送请求：3 + 5
    future = node.send_request(3, 5)

    # ⑥ 等结果：spin_until_future_complete 会阻塞直到服务端回复
    rclpy.spin_until_future_complete(node, future)

    # ⑦ 拿到响应
    response = future.result()
    node.get_logger().info(f'结果: 3 + 5 = {response.sum}')

    node.destroy_node()
    rclpy.shutdown()
```

### 逐行解释

```python
self.cli = self.create_client(AddTwoInts, 'add_two_ints')
```

和发布者/订阅者一个套路：服务类型 + 服务名。服务名必须和服务端一致，不然找不到。

```python
while not self.cli.wait_for_service(timeout_sec=1.0):
    self.get_logger().info('⏳ 等待服务端上线…')
```

服务端可能还没启动，`wait_for_service()` 会阻塞等待，直到服务端就绪。这是一种**健壮性**写法——先确认对方在线，再发请求。

```python
request = AddTwoInts.Request()
request.a = a
request.b = b
```

服务类型的 `Request` 子类用来构造请求。`AddTwoInts.Request` 有两个字段：`a` 和 `b`。

```python
self.future = self.cli.call_async(request)
```

`call_async()` 是异步调用——发完请求立刻返回一个 `future` 对象，不阻塞。想拿结果时再 `.result()`。

```python
rclpy.spin_until_future_complete(node, future)
```

这是关键一步：阻塞等待 future 完成（即服务端返回了响应）。等价于「等电话接通」。在此期间 ROS2 底层仍在处理网络消息，所以服务端能把响应送回来。

```
spin_until_future_complete  ≈  spin() + 等待某个 future 完成就停
```

> `spin_until_future_complete` 是服务客户端最常用的模式。也有同步版 `call()`，但它一调就死等，不推荐。

## 运行演示

打开**两个终端**：

```
终端1（服务端）：                       终端2（客户端）：
$ ros2 run                              $ ros2 run
  learning_service                       learning_service
  add_two_ints_server                     add_two_ints_client

🔧 服务端已启动，等待请求…               ⏳ 等待服务端上线…
收到: 3 + 5 = 8                          ✅ 服务端已连接
                                         结果: 3 + 5 = 8
```

**启动顺序无关紧要。** 客户端代码里有 `wait_for_service()`，服务端没起就每 1 秒重试一次，等到连上为止。先起客户端、再起服务端，一样能通。

运行完客户端就退出了，服务端继续等下一个请求。想再测一次？换个终端再跑一次客户端：

```bash
ros2 run learning_service add_two_ints_client        # 默认 3 + 5
ros2 run learning_service add_two_ints_client 10 20  # 10 + 20
```

## 常用命令

```bash
# 查看所有服务
ros2 service list                     # 列出所有服务

# 查看服务详情
ros2 service type /add_two_ints       # 查看服务使用的类型
# 输出: example_interfaces/srv/AddTwoInts

# 手动调用服务（测试神器，不用写客户端！）
ros2 service call /add_two_ints example_interfaces/srv/AddTwoInts "{a: 10, b: 20}"
# 输出: sum: 30
```

> `ros2 service call` 和 `ros2 topic pub` 一样是测试利器。不用写客户端就能调服务端，秒测。

## 为什么服务端不需要定时器

回顾一下：

| | 话题订阅者 | 服务端 |
|------|-----------|--------|
| 触发方式 | 发布者发消息时，回调被调用 | 客户端发请求时，回调被调用 |
| 需要定时器？ | 不需要 | 不需要 |
| 阻塞方式 | `rclpy.spin(node)` | `rclpy.spin(node)` |

两者都靠 `spin()` 循环在后台等待。区别在于：
- 话题：spin 收消息 → 调回调 → 继续 spin
- 服务：spin 收请求 → 调回调 → 返回响应 → 继续 spin

`spin()` 是 ROS2 的万能等待循环，话题、服务、定时器全都靠它驱动。

## 什么时候用话题，什么时候用服务

```
话题（Topic）适合：
  ✅ 传感器数据流（激光雷达、摄像头、IMU）
  ✅ 状态持续上报（电池电压、位姿、温度）
  ✅ 一（发）对多（收）的场景
  ✅ 不需要每次确认收没收到

服务（Service）适合：
  ✅ 一次性的查询（「当前温度多少？」）
  ✅ 配置变更（「把曝光调到 500」）
  ✅ 触发动作（「拍照」「复位」「校准」）
  ✅ 需要返回值的计算
```

经验法则：**数据流用话题，指令调用用服务。**

## 总结

```
服务通信 = 客户端 + 服务端 + 服务名

服务端三要素：
  ① create_service(服务类型, 服务名, 回调函数)
  ② 回调函数 callback(self, request, response) → 操作 response → return response
  ③ rclpy.spin(node) 等待请求

客户端四要素：
  ① create_client(服务类型, 服务名)
  ② wait_for_service() 等待服务端上线
  ③ 构造 Request → call_async(request) 发送
  ④ spin_until_future_complete() 等待响应 → future.result() 拿结果

核心规则：
  - 服务类型必须一致（AddTwoInts ↔ AddTwoInts）
  - 服务名必须一致（'add_two_ints' ↔ 'add_two_ints'）
  - 客户端发起请求，服务端返回响应
  - 一次调用 = 一次往返
  - ros2 service call 是最方便的测试方式
```

| 对比 | 客户端 Client | 服务端 Server |
|------|-------------|-------------|
| 角色 | 发起请求，等结果 | 等待请求，返回结果 |
| 创建方式 | `create_client()` | `create_service()` |
| 触发方式 | 主动 `call_async()` | 被动回调 `callback()` |
| 拿结果 | `future.result()` | `return response` |

话题通信是「广播」，服务通信是「打电话」。掌握这两种模式，ROS2 里 90% 的通信需求都能搞定。下一步：**动作（Action）**——带反馈的长时间任务，比如「从 A 开到 B，中间每分钟报告一次进度」。
